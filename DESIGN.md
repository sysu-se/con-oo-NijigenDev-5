## 对象设计说明

### 1. `Sudoku` / `Game` 的职责边界是什么？
* **Sudoku**：负责维护一个 9x9 的数字网格数据。它的职责边界是“保证每一次落子/获取都是针对此数据源”，并且负责数独规则的校验（计算行、列、宫中是否有冲突产生）。基本上就是记录盘面，提供操作或查询的接口。
* **Game**：聚焦于游戏状态的管理。它持有 `Sudoku`，记录用户操作的 `history`（存快照），并在用户请求撤销（Undo）或重做（Redo）时控制时空切换。它扮演了领域对象和展示组件间的接口。主要业务的逻辑都被收束在此处，不在UI组件中。

### 2. `Move` 是值对象还是实体对象？为什么？
`Move` 被设计为**值对象（Value Object）**。
因为它仅仅表达了在哪个坐标落什么值，如 `{row: 1, col: 1, value: 5}`。它不唯一，且它是不可变的。不同的 Move 对象即使是同一个坐标同一个值也仅仅意味着发生过同样的指令。它没有成为复杂核心对象的必要性，以轻量级传参即可。

### 3. history 中存储的是什么？为什么？
我采取了**快照**：
`history` 数组中存储的是**一系列完整的 `Sudoku` 对象**，而不单单是 `Move` 的动作差异序列。
这样做是因为容错率高和兼容性好。撤销仅是简单地将上一状态恢复回去，规避了复杂的操作。这是空间换取稳定性与简化。

### 4. 你的复制策略是什么？哪些地方需要深拷贝？
我的复制策略为：**涉及到`grid`时，不使用任何引用，所有数据均要求深拷贝。**
我使用数组展开结构实现了两层深拷贝 `grid.map(row => [...row])`。

深拷贝发生的必须节点如下：
1. **构造函数**：`createSudoku(input)` 时，避免外部通过最初传进来的 `input` 数组地址后门绕界更改。
2. **向UI组件传递**：`getGrid()` 时，防止 Svelte 的双向绑定或外界直接改写引用的当前有效状态。
3. **记录快照时**：在 `createGame` 每次往 `history` 写数据，以及触发对局克隆时 `sudoku.clone()` 必须深拷贝，如果不深拷贝会导致历史记录数组中的所有记录指针指向的是同一张棋盘（修改当前盘等于破坏所有历史记录）。

### 5. 你的序列化 / 反序列化设计是什么？
为了实现从内存运行态还原回持久化存储功能（或在网络传输），明确序列化设计的边界：

* **哪些字段会被序列化：**
  * `Sudoku` 对象：仅序列化 `grid`（9x9 的二维数组）。
  * `Game` 对象：只会序列化指针 `currentIndex` 以及包含完整时空记录的 `history` 数组（数组内递归调用其中每一个 `Sudoku` 对象的 `toJSON` 输出）。
* **哪些字段不会被序列化：**
  * 所有的方法，因为它们属于控制逻辑，不可能也无需 JSON 化。
  * 任何可由 `grid` 本身计算出的状态（例如：当前游戏是否胜利 `isSolved` 状态、当前盘面哪些格子有冲突 `getInvalidCells` 等），这类状态会在运行期生成。
* **恢复时如何重建对象（反序列化流程）：**
  * **对于 `Sudoku` 的重建**：暴露 `createSudokuFromJSON(json)`。它提取纯格式对象体里的 `grid` 数组，结合深拷贝将其重新注入到 `createSudoku()` 中。
  * **对于 `Game` 的重建**：借由 `createGameFromJSON(json)`。这会先通过 `map` 历遍其 `history` 数组序列，逐个调用 `createSudokuFromJSON` 生成一个个 `Sudoku` ；然后重建快照数组和保存的 `currentIndex`。使其拥有了和备份前一致的历史状态。

### 6. 你的外表化接口是什么？为什么这样设计？
 `toString()` 作为调试专用的外表化接口：
可以通过文本打印出矩阵（包含 `.` 与填入数字组成的字符串）。能让测试结果可见，而不依赖任何前端绘制逻辑。

## 领域对象接入 Svelte 响应式框架说明（本次作业）

### A. 领域对象如何被消费
1. **View 层直接消费的是什么？**
   View 层不直接消费原生的 `Game` 或 `Sudoku` 实例，而是消费一个 Store Adapter 适配层（对应 `domainStore` ）。导出代理的 `derived store`（如重构后的 `grid`, `userGrid`, `invalidCells`），供 View 消费。
2. **View 层拿到的数据是什么？**
   拿到的是暴露出来的 `writable`。包含：`grid`（当前局面）、`initialGrid`（初始只读题目局面）、`invalidCells`（格式化为字符串数组的坐标）、`isSolved`（是否胜利）、`canUndo`、`canRedo` 等状态。
3. **用户操作如何进入领域对象？**
   用户在组件（如点击键盘时）发出操作请求,先进入原有的,已经被代理的 `grid`、 `userGrid`，调用 `domainStore.guess(y, x, value)` 或 `domainStore.undo()` 等适配命令，适配命令再去调用内部封装的纯 `Game` 实例的方法 `gameInstance.guess()`。
4. **领域对象变化后，Svelte 为什么会更新？**
   在 `domainStore` 内部每次调用完领域实例方法后，都会显式调用一个 `sync()` 方法。这个方法会把 `Game` 实例中最新的变量取出来，并通过 Svelte store 的 `set()` 方法将其覆盖赋值回去。这契合了 Svelte 基于赋值的响应机制，成功触发 UI 重新渲染。

### B. 响应式机制说明
1. **你依赖的是什么机制？**
   我主要依赖 Svelte 的 `store` (`writable` / `derived`) 和组件中的 `$store` 语法糖。Svelte 编译器遇到 `$store` 时，会自动绑定 `subscribe` 并在 store `set` 新值时隐式触发更新。
2. **哪些数据是响应式暴露给 UI 的？**
   `grid`、`initialGrid`、`invalidCells`、胜利标识以及能否 Undo/Redo 的标识等，这些界面渲染必需的数据都被囊括进了响应式对象。
3. **哪些状态留在领域对象内部？**
   真实的 `sudoku` 和 `Game` 的类实例、 `history` 快照栈、当前指针 `currentIndex` 等承担内部逻辑的结构封闭在对象内部。
4. **如果不用这个方案，而是直接 mutate 内部对象，会出现什么问题？**
   Svelte 的响应式是在编译时基于赋值语句（`=`）建立的。如果不用 Store `set` 或重新赋值，在组件中直接执行 `myGame.grid[0][0] = 5` 时，`myGame` 这个变量本身的引用没有改变，Svelte 无法捕捉到深层的数据变动，就会导致出现底层数据已经改了，但界面数据仍然不变的问题。

### C. 改进说明
1. **相比 HW1，改进了什么？**
   * **安全性**：`Game` 暴露的获取接口被改为了深拷贝 `clone()` 返回（防止外部改动绕开 `history`）；闭包内去除了不规范的 `this` 。
   * **规则保证**：在 `createGame` 中特地引入了 `initialSudoku` 快照，严格限制任何意图对“初始题目区域”进行的 `guess()` 赋值。也可供 UI 层区分初始数字与玩家输入。
   * **增强鲁棒性**：修复了胜利判定对空格（`0`与`null`）的区分。
   * **接管 Hint 功能**：加入了 `applyHint` 逻辑，应用 hint 时，将其永久固化进 `initialSudoku` ，禁止玩家修改，也让历史所有记录都包含这个变更，不可撤销。
   * **接入框架**：增加了符合 Svelte 要求的适配层 `domainStore`。
     * 接管了游戏初始化、用户输入、Undo/Redo、hint、胜利判定、以及 UI 需要的状态计算（如 `invalidCells`）。
2. **为什么 HW1 中的做法不足以支撑真实接入？**
   HW1 中对象内部再怎么调用自身的方法，UI 框架都不可能感知到状态刷新。
3. **你的新设计有哪些 trade-off？**
   * **最小改动**：我选择在 `grid.js` 中保留了 `userGrid` 和 `grid` 这两个遗存的 Store，让它们通过 `derived` 代理指向底层的 `domainStore`，并且与原先行为保持一致。
     * *优势*：最小化了重构的范围，使得 UI 组件层（如`keyboard` 等，以及 svelte 组件）无需修改。
     * *代价*：代码中多出了一层间接的代理层，且使得 View 层依旧带着一些业务逻辑，如 hint 使用个数的计算（虽然这个逻辑也可以放在 `domainStore` 中，但我觉得它更像是 UI 需要的状态，而非领域对象的核心职责）。不够简洁。
